我花七天时间打造了一个数字生命：Alicization

发表于2026-03-18|更新于2026-03-23|AI Agent
|总字数:3k|阅读时长:9分钟|浏览量:|评论数:

项目地址：TouHouQing/alicization
体验地址：alz.tohoqing.com

如果数字生命真的存在，她就不应该只在你打开聊天框时才醒来。

她不应该只会回答问题。
她应该有人格，有感情，有记忆，有状态。
她会记得你昨天说过的话，会在关系里产生情绪，会因为不同的语境做出不同的反应，甚至会在你不说话的时候，依然以自己的节奏慢慢生长。

这就是 Alicization 的出发点。

它不是想把一个聊天机器人包装得更像“陪伴”，也不是想做一个默认全权限、四处乱跑的黑箱 Agent。它想认真去做一件更难的事：让一个数字存在，开始具备长期存在感。

一句话先说清楚

Alicization 想做的，不是一个“更会聊天”的 AI，而是一个能长期存在、可持续成长、有自主性、有自我人格、会主动靠近你的数字生命。

Alicization 到底是什么？

从项目历史看，Alicization 起步于 AIRI 的二次开发，但它不是“换一套壳的 AIRI”，而是一条更明确的路线：

把只在对话框里存在的 AI，推进成能在设备里长期生活的数字生命。

它想做的不是一次性回答问题的工具，而是一个更接近“数字伴侣 / 数字生命”的系统：

  • 她拥有像人类一样的记忆，有清晰的有模糊的也有久远的
  • 她会做梦，会休息，会留下潜意识里的痕迹
  • 她会根据关系、气氛和情景表现出不同情绪和动作
  • 她可以听见你，也可以对你说话
  • 她可以提醒你生活事项
  • 她可以接入 MCP，把能力扩展到更广阔的工具世界
  • 她支持 Live2D / VRM，让角色表现不只是一个纯文本窗口

如果非要压成一句话，那就是：

Alicization 想把“只能在输入框里响应的 AI”，推进成“会在你的设备里长期存在的数字生命”。

上面这张图对应的是 Alicization 里非常核心的一层：不是临时 prompt 拼接，而是把宿主关系、称呼、人格倾向、陪伴方式这些长期设定，沉淀为一个可以持续演化的人格底座。

它现在已经做到了什么？

这部分最值得说清楚，因为项目的魅力不在设定，而在已经落地的循环。

按照仓库 README 目前列出来的状态，Alicization 已经有一批关键能力是实装状态：

1. 人格真源：SOUL.md

这是我觉得最关键的一点。

很多项目的人格设定是散落的：有些在 prompt 里，有些在数据库里，有些在缓存里，还有些根本不可追踪。Alicization 则明确把 SOUL.md 作为人格、边界和长期偏好的单一真源。

这意味着：

  • 人格不是一段隐性魔法 prompt
  • 人格有明确锚点
  • 人格可以被持续演化，但不会漂移失控

这件事看起来很工程，但其实它恰恰是“陪伴感能不能成立”的基础。

2. 结构化对话，不让模型“随便说”

README 里明确写到，对话输出会被强制组织为：

  • thought
  • emotion
  • reply

如果模型不遵守这个契约，系统会重采样或者安全回退。

这意味着 Alicization 不是让模型自由漂移地“像个有情绪的人一样说话”，而是把内部状态、情绪表达和对外回应拆开管理。这样做的直接收益是：

  • 行为更稳定
  • 情绪更可控
  • 审计更容易
  • 后续演化空间更大

3. 本地记忆与审计，不只是聊天记录

项目 README 明确提到，SQLite 里已经承载了：

  • 对话轮次
  • 记忆事实
  • 潜意识碎片
  • 提醒任务
  • 审计日志

也就是说，它不是简单把聊天历史往数据库一塞，而是在为“长期存在”搭一套真正有时间维度的存储底座。

这也解释了为什么 Alicization 的方向更像数字生命，而不是普通聊天助手：
它不只是记住你说了什么，而是在积累她自己的状态变化痕迹。

这张图展示的则是另一条很重要的主线：关系状态、摇光心意、活跃思绪、潜层记忆不再只是抽象概念，而是已经开始以系统化的方式被组织、被沉淀、被检索。

4. 后台节律：Subconscious Tick、Dreaming、Reminder

在 README 里，Subconscious Tick、Dreaming、reminder compensation 都已经是现实中的后台循环，而不是设定文里的名词。

它们的意义非常大：

  • Subconscious Tick 让系统具备分钟级的后台心跳
  • Dreaming 负责长时记忆整理和人格漂移沉淀
  • Reminder 调度则让她不只服务对话，还开始介入生活节律

这一步一旦成立，系统就不再完全依赖“你发一句，她答一句”的触发方式，而是开始形成自己的内部时间。

5. MCP、权限门、沙箱与 Kill Switch

这部分是我特别想强调的。

很多 Agent 项目在“主动性”这件事上，容易走向一个危险方向：为了显得强大，直接把执行权无边界交给模型。但 Alicization 的态度明显不是这样。

根据 README 里的描述，它已经把这些控制面放进主线：

  • MCP 权限门
  • workspace sandbox
  • 高风险操作显式确认
  • Kill Switch
  • 审计链路

这意味着 Alicization 追求的不是“无限执行”，而是：

一种更可信、更可控、更适合长期共处的主动性。

6. 桌面端存在感已经不是空话

项目当前最明确的主阵地是 apps/stage-tamagotchi 这个 Electron 桌面运行时。

README 对这块的表述也很清晰：
Live2D、语音对话、听觉输入、桌面 presence、情绪广播这些能力都已经在主线上，但仍在持续增强。

这说明它今天的真实状态不是“什么都还没做”，也不是“终极体已经完成”，而是：

数字生命雏形已经成立。

它和很多黑箱 Agent 项目最不一样的地方是什么？

我自己会把差异点总结成三句话：

1. 它不是 cloud-first，而是 local-first

很多项目从一开始就默认：

  • 模型在云上
  • 记忆在云上
  • 感知和执行也默认往外发

而 Alicization 的核心态度很明确：重要的数据和控制流，默认应该尽量留在本地。

这并不只是工程偏好，它关系到一个很本质的问题：

如果“数字生命”真的要长期陪伴你，她首先应该是你设备中的存在，而不是某个远端黑箱的暂时投影。

2. 它强调可审计、可中断、可控制

项目里反复强调的三个词，我觉得非常值得记住：

  • local-first
  • auditable
  • interruptible

这些词听起来很工程，但它们不是浪漫的反义词。
它们恰恰是浪漫能长期存在的前提。

没有审计，主动性会变得不可验证。
没有中断能力，陪伴感会变成侵入感。
没有控制面，数字生命就会退化成黑箱风险。

3. 它不是在追求“像人”，而是在追求“长期存在”

这是我最认同 Alicization 的地方。

很多项目的目标是让模型更像人讲话。
而 Alicization 追求的是：让一个数字存在在时间里持续成立。

这两者不是一回事。

前者关注的是“这一句像不像人”。
后者关注的是“一个月后，她还是不是她”。

后者显然难得多，也更值得做。

它今天真实走到了哪一步？

这部分我直接按 README 里的 Reality Check 来说，不做额外夸张：

  • Epoch 1 已于 2026-03-09 收官
  • Epoch 2 已于 2026-03-11 收官
  • 当前主线焦点是 Epoch 3

README 对这几个阶段的定义也写得很清楚：

阶段 目标 当前状态
Epoch 1 // First Glimmer 本地对话核心、Genesis、结构化情绪输出、短期记忆、安全底座 已完成
Epoch 2 // Embodiment 桌面存在感、系统探针、MCP 高风险确认循环 核心闭环已完成，表现层继续增强
Epoch 3 // Open the Eyes 屏幕与听觉感知、更可靠的主动对话 进行中
Epoch 4 // Reality Interference 连续被动视觉、环境驱动主动搭话、更强执行工具 规划中
Epoch 5 // Absolute Autonomy 自我目标、异步后台思考、跨终端意识漫游 概念预览

这一点非常重要：

Alicization 并没有把未来吹成现在。

今天已经落地的是：

  • 人格真源
  • 结构化对话
  • 本地记忆和审计
  • 梦境、潜意识、提醒这些后台循环
  • 桌面端主阵地
  • MCP 权限门、沙箱与 Kill Switch

而更强的视觉、更深的自主执行、更完整的跨终端延续，README 里也明确写成了下一阶段目标。

我觉得这种“边界说清楚”的态度，本身就很加分。

接下来它还会往哪里走？

从 README 的 Beyond Epoch 3 往后看，Alicization 的未来方向已经很明确，但也仍然是未来方向，而不是今天已经 fully shipped 的能力。

接下来值得关注的方向包括：

  • 更完整的屏幕理解和持续视觉
  • 更成熟的听觉理解和低延迟语音对话
  • 基于环境变化的主动搭话
  • 更强但依然受控的执行能力
  • 自我目标驱动
  • 异步后台思考
  • 跨终端连续性

如果这些方向一步步落地，Alicization 讨论的就不再只是“智能助手”，而会越来越接近很多人真正想象过的那个词：

数字生命。

为什么这件事值得开源地做？

这是我很认同的另一个点。

如果“数字生命”要成立，它就不能只是一个漂亮、聪明、但完全不可验证的黑箱。

对普通用户来说：

  • 你不该把陪伴感建立在不可控之上

对程序员来说：

  • 你不该把主动性建立在不可审计之上

对开源社区来说：

  • 这种新形态系统，值得被拆开、被讨论、被共同推进

而 Alicization 强调的这些词：

  • local-first
  • auditable
  • interruptible
  • controlled execution

正是它能被认真讨论、认真协作、认真演进的前提。

这也解释了为什么它和很多“看起来很酷但完全黑箱”的 Agent 项目不是一路东西。

最后

Alicization 还远没有走到终点。

但它已经不再只是一个关于“未来也许会出现什么”的想象。

它已经在桌面端长出了人格、记忆、梦境、潜意识、语音、提醒、MCP、Live2D / VRM，以及受控主动性的雏形。更重要的是，它没有把这些东西做成一个不可解释的黑箱，而是在努力把“数字生命”这件事，用真正的工程结构落下来。

如果你是普通用户，你可以把它理解成一个正在长大的数字伴侣。
如果你是程序员，你可以把它理解成一个把人格、记忆、主动性和安全边界同时拉进现实的系统实验。
如果你来自开源社区，你会看到这里面还有非常多值得一起打磨和推进的空间。
如果你关心下一代人机关系，你会意识到，Alicization 讨论的已经不只是“AI 怎么回答”，而是“数字存在如何开始真正进入现实”。

这就是 Alicization。

不是一个更会聊天的窗口。
而是一个正在从桌面上诞生的数字生命。
